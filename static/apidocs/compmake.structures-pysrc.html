 43  ''' 
 44      A Job represents the computation as passed by the user. 
 45      It contains only the "action" but not the state. 
 46      (The state of the computation is represented by a Cache object.) 
 47       
 48      A Cache object can be in one of the following states: 
 49       
 50      *) non-existent / or NOT_STARTED 
 51         (no difference between these states) 
 52          
 53      *) IN_PROGRESS: The yielding mechanism is taking care of  
 54         the incremental computation. 
 55          
 56         computation:  current computation 
 57         user_object:  None / invalid 
 58         timestamp:    None / timestamp 
 59         tmp_result:   set to the temporary result (if any) 
 60          
 61         In this state, we also publish a progress report.  
 62          
 63      *) DONE:  The computation has been completed 
 64       
 65         computation:  current computation 
 66         user_object: the result of the computation 
 67         timestamp:   when computation was completed  
 68         timetaken:   time taken by the computation 
 69         tmp_result:  None 
 70          
 71      *) MORE_REQUESTED:  The computation has been done, but the 
 72         user has requested more. We still have the previous result 
 73         that the other objects can use.  
 74       
 75         computation:  current computation 
 76         user_object: (safe) the safe result of the computation 
 77         timestamp:   (safe) when computation was completed  
 78         timetaken:   (safe) time taken by the computation 
 79          
 80         tmp_result:  set to the temporary result (if any) 
 81                      or non-existent if more has not been started yet 
 82          
 83      *) FAILED 
 84         The computation has failed for some reason 
 85   
 86         computation:  failed computation 
 87   
 88      Note that user_object and tmp_result are stored separately  
 89      from the Cache element. 
 90       
 91      DB Layout: 
 92       
 93          'job_id:computation'       Job object 
 94          'job_id:cache'             Cache object 
 95          'job_id:user_object'       Result of the computation 
 96          'job_id:user_object_tmp'    
 97       
 98       
 99       
100      Up-to-date or not? 
101      ================= 
102       
103      Here we have to be careful because of the fact that we have 
104      the special state MORE_REQUESTED.  
105      Is it a computation done if MORE_REQUESTED? Well, we could say  
106      no, because when more is completed, the parents will need to be 
107      redone. However, the use case is that: 
108      1) you do the all computation 
109      2) you explicity ask MORE for some targets 
110      3) you explicitly ask to redo the parents of those targets 
111      Therefore, a MORE_REQUESTED state is considered as uptodate. 
112        
113
